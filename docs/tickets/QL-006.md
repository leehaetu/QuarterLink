## Goal

Auth, tenant, audit, secrets, and HMRC access architecture.

Decide the minimum architecture needed before real HMRC sandbox work so production access is not blocked later by weak security, token handling, audit, consent, or fraud-prevention design.

This is primarily an architecture/specification ticket. It must define the implementation boundaries for the next HMRC sandbox work without prematurely building authentication, database persistence, HMRC OAuth, or HMRC API calls.

## Required architecture decisions

The architecture document must decide and record the minimum approach for the first individual-user sandbox-readiness path.

It must cover:

### 1. User and tenant model

- first slice user type: individual taxpayer;
- tenant boundary for a single individual user;
- how future practice/accountant tenancy must not pollute the first slice;
- how tenant isolation will later work when multiple users and practices exist;
- what must be tenant-scoped from day one.

### 2. Auth and session model

- recommended MVP auth approach;
- session handling assumptions;
- MFA/2FA stance for production-readiness;
- what can be deferred until after sandbox foundation;
- what must exist before any real HMRC OAuth token is stored.

### 3. HMRC OAuth and token handling

- server-side-only token handling;
- no HMRC tokens in browser-accessible state;
- token encryption-at-rest expectations;
- refresh-token handling;
- token revocation/disconnect behaviour;
- audit events for connect, refresh, failure, disconnect;
- environment separation between local, sandbox, and production.

### 4. Secrets and environment variables

- required secret categories;
- storage location expectations;
- rotation expectations;
- what must never be committed;
- local development rules;
- Railway/deployment environment assumptions if relevant;
- fail-fast checks needed before QL-007.

### 5. Audit events

Define a practical audit-event catalogue for the MVP.

For each event, include:

- event name;
- trigger;
- actor;
- tenant scope;
- taxpayer/income-source scope if relevant;
- required metadata;
- sensitive values that must not be stored;
- whether the event is required before sandbox testing or can wait.

### 6. Consent and declaration records

Define what needs to be captured before a quarterly update is eventually sent, including:

- user consent to connect to HMRC;
- authority/permission to send;
- declaration acknowledgement;
- evidence-review acknowledgement;
- timestamping;
- versioned wording.

### 7. Fraud prevention header architecture

Define how QuarterLink will collect, derive, and transmit HMRC fraud-prevention-header data.

The fraud-prevention-header architecture must cover:

- which data is collected client-side;
- which data is derived server-side;
- which data must never be spoofed by trusted client input alone;
- how headers are assembled server-side;
- logging rules for fraud header data;
- redaction rules;
- local/sandbox/production differences;
- fallback/error behaviour when required data is missing;
- what must be implemented before QL-007 can make real HMRC sandbox calls.

### 8. What must never be exposed to the browser

List sensitive data that must not be sent to client components, local storage, console logs, analytics, or downloadable evidence bundles.

Include at least:

- HMRC client secret;
- HMRC access token;
- HMRC refresh token;
- encryption keys;
- session signing secrets;
- raw secret environment variables;
- unredacted fraud-prevention metadata where inappropriate;
- internal support/admin notes;
- unrelated tenant/taxpayer data.

### 9. QL-007 readiness gate

Define the exact checklist that must pass before QL-007 starts real HMRC sandbox foundation work.

The checklist must include at least:

- auth/session decision accepted;
- tenant boundary accepted;
- token-storage approach accepted;
- secrets approach accepted;
- fraud-prevention-header architecture accepted;
- audit-event minimum accepted;
- consent/declaration minimum accepted;
- HMRC guidance verification items recorded;
- no HMRC recognised/production-ready claims added.
