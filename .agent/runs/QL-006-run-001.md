## Architecture Decisions Made

- First sandbox-readiness user model is one individual taxpayer, one individual tenant, one taxpayer profile, one self-employment income source, and one HMRC connection per taxpayer profile.
- Practice, agent, billing, delegated-user, and platform-admin features remain deferred and must not pollute the first individual-user tenant boundary.
- HMRC tokens, authorisation codes, client secrets, PKCE verifiers, encryption keys, session secrets, and raw environment secrets must never be exposed to the browser.
- HMRC OAuth must be started, validated, exchanged, refreshed, revoked, and audited server-side.
- Real HMRC OAuth token storage must wait until authenticated user, tenant, server-verifiable session, audit logging, and token encryption-at-rest exist.
- Secrets must be managed through ignored local environment files or deployment-provider secret storage, never committed.
- The audit model is append-only, tenant-scoped, server-written, redacted by default, and required before sandbox calls.
- Consent and declaration records must be versioned, timestamped, linked to the tenant/taxpayer/income source/period, and human-reviewed for wording.
- Fraud prevention header architecture uses `WEB_APP_VIA_SERVER`.
- Client-collected fraud data is limited to browser/device/screen/window/timezone/MFA prompt metadata; IP, port, forwarded chain, identity, product, version, scopes, and token status are server-derived.
- Missing required fraud-prevention-header data blocks HMRC calls by default unless a documented missing-data handling approach agreed with HMRC exists.
- Test Fraud Prevention Headers API use is required before Income Tax MTD sandbox API calls.
